## Implementation Code

```java
import java.util.Map;
import java.util.LinkedHashMap;

@Command(
    name = "pg-ssh-config",
    defaultValueProvider = PropertiesDefaultProvider.class,
    mixinStandardHelpOptions = true
)
public class PgSshConfigurer implements Runnable {

    @Option(
        names = "--set",
        description = "PostgreSQL config settings (key=value pairs separated by semicolons)",
        descriptionKey = "set",
        split = ";",
        splitSynopsisLabel = ";"
    )
    Map<String, String> desiredProperties = new LinkedHashMap<>();

    @Override
    public void run() {
        // desiredProperties is already a Map<String, String>
        // No parsing needed!
        
        for (Map.Entry<String, String> entry : desiredProperties.entrySet()) {
            String key = entry.getKey();
            String value = entry.getValue();
            
            // Reconcile each property
            configFileManager.reconcileProperty(configFilePath, key, value);
        }
    }
}
```

## Escaping & Special Characters

### Values with Spaces (requires quoting on shell)
```bash
--set "search_path='public, schema2';timezone='UTC'"
```

### Values with Semicolons (need escaping)
```bash
# Use backslash to escape semicolons in values
--set "name=value\;with\;semicolons"
```

### Values with Equals Signs
```bash
--set "equation=2+2=4"  # Works fine, equals only matters for key=value split
```

## Delimiter Character Decision: Semicolon (`;`)

Why semicolon is optimal for PostgreSQL:
- ✅ Rarely appears in PostgreSQL config values
- ✅ Commonly used in config files for separation (SQL statements, bashrc, etc.)
- ✅ Shell-safe (doesn't trigger redirects like `|` or command chains)
- ✅ Consistent with semicolon-terminated SQL statements
- ❌ Comma: Conflicts with PostgreSQL array syntax `foo='{a,b,c}'`
- ❌ Pipe: Reserved by shell for process piping
- ❌ Colon: Contains too many RFC/YAML meanings
